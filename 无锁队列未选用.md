不是"不选择"，是实现有 bug：

内存序 bug：write_pos_.store(w+1, release) 在 buf_[w&MASK] = std::move(cmd) 之后。但 release 语义只保证之前的写入对后续 acquire 可见。问题在于：PendingCommand 内含 lstl::vector，move 之后源 vector 变空。如果 consumer 在 move 完成之前读到 write_pos_（CPU 乱序），就会读到半移动状态的 vector → crash。

没有反压机制：队列满时自旋 yield()。当 Dict 300K 条目 rehash 拖慢存储线程时，队列堆积，IO 线程全部自旋抢 CPU，存储线程更慢，恶性循环。

Mutex 版本实际上够快：Mutex::Lock + vector::swap 总共 ~100ns。对比 Dict::insert(~500ns) + eventfd(~1us) + fiber yield(~5us)，队列开销占比 < 2%。不值得为这 2% 冒稳定性风险。

正确的无锁队列需要：

内存序用 acquire/release + 必要时 seq_cst fence
Consumer 批量 drain 时保证可见性
反压用 semaphore 或 eventfd 阻塞等待，而非自旋
